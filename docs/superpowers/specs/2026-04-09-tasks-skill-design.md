# Skill `tasks` — дизайн

**Дата:** 2026-04-09
**Статус:** draft → on review

## 1. Назначение

Лёгкий менеджер задач проекта, живущий в `TASKS.md` в корне репозитория. Скилл предоставляет две команды и после их отработки «забывается» — никаких фоновых статусов, отслеживания прогресса, напоминаний.

**Команды:**

- `/add <формулировка>` — оценить задачу по матрице Эйзенхауэра (важность × срочность), подтвердить оценку у пользователя, положить строку в нужный квадрант `TASKS.md`.
- `/next` — предложить топ-3 следующих задач, перенести выбранную в `Done` **немедленно**, предложить запустить `superpowers:brainstorming` для продумывания реализации.

**Принцип:** скилл отвечает только за учёт. Как только задача выбрана через `/next`, она сразу уходит в `Done` — дальше ответственность за её реализацию лежит на пользователе. Это сделано осознанно, чтобы Claude не держал в контексте «не забыть в конце закрыть задачу».

## 2. Формат `TASKS.md`

Файл в корне проекта с фиксированной структурой:

```markdown
# Tasks

## Важно и срочно

- [важность: high, срочность: high] Починить падение pipeline при пустом .srt #tech #bug
- [важность: high, срочность: high] AnkiConnect отваливается после обновления Anki #tech #bug

## Важно, не срочно

- [важность: high, срочность: med] Вынести отбор target'ов в отдельный use case #tech #refactor
- [важность: med, срочность: low] Добавить поддержку .vtt субтитров #product

## Не важно, срочно

- [важность: low, срочность: high] Обновить README — скриншоты устарели перед демо #docs

## Не важно, не срочно

- [важность: low, срочность: low] Поиграться со stanza вместо spaCy #tech #experiment

## Done

- ~~[важность: high, срочность: high] Настроить docker-compose для прод-сборки #tech~~ (2026-04-05)
- ~~[важность: med, срочность: med] Миграция кнопки превью карточки #product~~ (2026-04-07)
```

**Правила формата:**

- Все 4 рабочих квадранта + секция `Done` присутствуют всегда, даже пустые — файл должен быть предсказуем для скилла.
- Задача = одна строка вида `- [важность: <level>, срочность: <level>] <текст> #tag1 #tag2`, где `<level>` ∈ {`high`, `med`, `low`}.
- Префикс `[важность: X, срочность: Y]` обязателен. Квадрант — это **огрубление** оценки (см. матрицу, §3), префикс хранит полную детализацию для истории и возможной сортировки. Без префикса теряется различие `high` vs `med` внутри одного квадранта.
- Теги опциональны — 0, 1 или несколько. Стандартные теги: `#tech`, `#product`, `#bug`, `#refactor`, `#docs`, `#experiment`. Дополнительные свободные теги допустимы.
- Новые задачи добавляются **в конец** квадранта (новые снизу).
- При переносе в `Done` задача оборачивается в `~~...~~` (GFM strikethrough), в конце строки добавляется `(YYYY-MM-DD)` — дата завершения. Префикс сохраняется внутри strikethrough.
- Секция `Done` растёт вниз бесконечно, скилл её не чистит.

## 3. Матрица приоритизации

Используется матрица Эйзенхауэра: `важность (high/med/low) × срочность (high/med/low)`.

Квадрант вычисляется из комбинации:

| Важность | Срочность | Квадрант |
|---|---|---|
| `high` или `med` | `high` | Важно и срочно |
| `high` или `med` | `med` или `low` | Важно, не срочно |
| `low` | `high` | Не важно, срочно |
| `low` | `med` или `low` | Не важно, не срочно |

**Почему `med` на «важной» стороне:** классический Эйзенхауэр бинарен («важно/не важно»), но мы держим три уровня для нюанса. Только явно низкая значимость (`low`) попадает в «не важно». `med` — это «важно, но не критично», его естественное место — Q1/Q2. Различие `high` vs `med` внутри одного квадранта не теряется, т.к. сохраняется в префиксе `[важность: X, срочность: Y]` и может быть использовано для сортировки или как справка.

**Как оценивает Claude:**

- **Важность** — влияет ли задача на ключевые цели проекта. Claude учитывает `CLAUDE.md` проекта (если есть) и формулировку задачи. `high` = напрямую бьёт по core-value или core-workflow. `med` = заметно улучшает продукт/devex/архитектуру, но не блокирует. `low` = эксперимент, полировка, «когда-нибудь».
- **Срочность** — есть ли временное давление. Признаки: явные маркеры «горит», «блокирует», «прод-баг», упоминания дедлайнов, внешних встреч/демо. `high` = блокер или жёсткий внешний дедлайн. `med` = хочется скоро, но без жёсткого срока. `low` = нет временного давления.

## 4. Поведение `/add <формулировка>`

### Шаг 1. Проверка файла

Если `TASKS.md` в корне проекта отсутствует — создать его с пустым каркасом (заголовок `# Tasks`, все 4 квадранта и `Done` без задач).

### Шаг 2. Проверка ясности формулировки

Claude оценивает, достаточно ли в формулировке информации для осмысленной оценки важности и срочности.

**Признаки недостаточной ясности:**

- Слишком коротко или общо: `/add починить баг`, `/add рефакторинг`, `/add добавить фичу`.
- Непонятно, о каком компоненте/файле/сценарии речь.
- Неясно, блокер ли это или «хорошо бы когда-нибудь».
- Непонятен контекст: баг в проде или гипотетический edge case.

Если формулировка размытая — Claude задаёт уточняющие вопросы **по одному**:

- Close-ended вопросы (например, «блокирует сейчас или можно отложить?») задаются через `AskUserQuestion` с готовыми вариантами.
- Open-ended вопросы, требующие развёрнутого ответа («что именно ломается?»), задаются обычным текстом.

После сбора достаточного контекста Claude **переформулирует задачу** в более конкретную и показывает через `AskUserQuestion`: «Переформулирую: *<новая формулировка>*. Ок?» с вариантами `Ок` / `Ещё поправить`. Цикл до подтверждения.

### Шаг 3. Оценка

**Главное правило обработки текста задачи:**

- **Переформулировать можно и часто полезно** — привести к связному виду, поправить грамматику, сделать фразы чётче. Пользователь может написать коряво или в потоке мысли — причесать это хорошо.
- **Сокращать и терять контекст — категорически нельзя.** Если пользователь прислал задачу с описанием «что ломается, где (файлы/компоненты/endpoints), почему, предлагаемый фикс, нюансы» — весь этот контекст должен оказаться в итоговой записи. Claude НЕ схлопывает задачу в короткий заголовок, НЕ выбрасывает упоминания файлов, таблиц БД, функций, причин инцидентов, альтернатив.
- **Самопроверка:** если из переформулированной версии нельзя понять «что конкретно делать и почему», а из исходника — можно, значит контекст потерян. Claude возвращает его.
- **Что можно отбросить как шум:** нумерационные префиксы в начале формулировки (`#11 — `, `[1]`, `Задача 1:`), явные инструкции скиллу («добавь в tasks», «запиши задачу», «нужно добавить»).
- Если оригинал уже хорошо сформулирован — Claude сохраняет дословно без попыток «улучшить ради улучшения».

Claude проставляет метаданные:

- Важность (`high` / `med` / `low`)
- Срочность (`high` / `med` / `low`)
- Теги (из стандартного набора + свободные, если явно напрашиваются)

Вычисляет квадрант по таблице из раздела 3.

### Шаг 4. Подтверждение

Claude одним сообщением показывает результат оценки в фиксированном формате и спрашивает через `AskUserQuestion`.

**Формат вывода (обязательный):**

```markdown
**Задача <N>/<total>** · _<квадрант>_

> <полный текст задачи, как будет записан в файл — со всеми абзацами, без сокращений>

- **Важность:** `<level>` — <краткое обоснование>
- **Срочность:** `<level>` — <краткое обоснование>
- **Теги:** `#tag1` `#tag2`
```

Заголовок `**Задача <N>/<total>**` показывается всегда: для одиночной задачи — `Задача 1/1`, для множественного `/add` — `Задача 1/4`, `Задача 2/4` и т.д. Текст задачи выводится **полностью** в blockquote, даже если на несколько абзацев. Это гарантирует, что пользователь видит ровно то, что будет записано в файл.

Варианты ответа в `AskUserQuestion`: `Ок, записать` / `Поправить`.

Если `Поправить` — Claude переходит в свободный диалог, принимает правки в произвольной форме («срочность low», «убери #bug, добавь #refactor», «это в product, не tech», «сократи формулировку», «добавь упоминание X»), применяет их, показывает обновлённую карточку в том же формате и снова спрашивает через `AskUserQuestion`. Цикл до подтверждения.

### Шаг 5. Запись

Claude дописывает задачу **в конец** нужного квадранта, сохраняя полный текст как он был в подтверждённом варианте (без дополнительных сокращений).

**Формат записи:**

- Однострочная задача: `- [важность: X, срочность: Y] <полный текст> #tag1 #tag2`.
- Многострочная задача: первая строка `- [важность: X, срочность: Y] <первая смысловая фраза>`, последующие строки и абзацы индентируются двумя пробелами (markdown-list continuation). Между абзацами внутри задачи — пустая строка. Теги ставятся в конце **последней** строки блока.
- Между блоками задач в одном квадранте — одна пустая строка.

**Подтверждение пользователю:**

- Если обрабатывалась одна задача — одна строка: `Добавлено в «<название квадранта>».`
- Если обрабатывалось несколько задач (через Шаг 0) — Claude не отвечает после каждой, а выводит сводку после обработки всех:

  ```markdown
  **Добавлено:**

  1. «<первая строка задачи 1>» → _<квадрант>_
  2. «<первая строка задачи 2>» → _<квадрант>_
  ```

Скилл завершается.

## 5. Поведение `/next`

### Шаг 1. Проверка файла

- Если `TASKS.md` отсутствует — ответить: `В проекте нет задач. Добавь первую через /add <формулировка>.` и завершиться. **Файл не создавать.**
- Если `TASKS.md` есть, но все 4 рабочих квадранта пусты (независимо от содержимого `Done`) — ответить: `Все задачи завершены. Добавь новые через /add.` и завершиться.

### Шаг 2. Сбор кандидатов

Claude собирает до 3 задач **строго по порядку квадрантов**:

1. `Важно и срочно`
2. `Важно, не срочно`
3. `Не важно, срочно`
4. `Не важно, не срочно`

Внутри квадранта — в порядке, в котором задачи записаны в файле (сверху вниз). Набирает первые 3, какие встретит. Не «перемешивает» между квадрантами и не учитывает контекст репы, теги или последние коммиты — логика детерминированная.

### Шаг 3. Предложение пользователю

Claude показывает кандидатов и спрашивает через `AskUserQuestion`. Каждая задача — отдельный вариант ответа с пометкой квадранта. Дополнительный вариант — `Другое (вручную)`.

Для каждой задачи в preview показывается **только первая строка** блока (после префикса оценки). Многострочный контекст в preview НЕ вставляется, чтобы сообщение оставалось компактным и читаемым — пользователь знает свои задачи и узнаёт их по первой фразе.

Пример сообщения:

```markdown
**Следующие задачи:**

1. Починить падение pipeline при пустом .srt · _Важно и срочно_ · `#tech #bug`
2. AnkiConnect отваливается после обновления Anki · _Важно и срочно_ · `#tech #bug`
3. Вынести отбор target'ов в отдельный use case · _Важно, не срочно_ · `#tech #refactor`
```

**Вариант `Другое (вручную)`** — Claude просит пользователя назвать задачу свободной формулировкой, ищет её в `TASKS.md` по нечёткому совпадению (подстрока, игнорируя регистр и знаки препинания, поиск и в первой строке блока, и в многострочном контексте). Если не находит — переспрашивает один раз; если снова нет — отвечает, что не смог найти, и предлагает выбрать из предложенных трёх.

### Шаг 4. Перенос в `Done`

Как только пользователь выбрал задачу — **немедленно**, до обсуждения реализации:

1. Claude определяет границы **блока задачи**. Блок начинается со строки `- [важность: X, срочность: Y] ...` и включает все последующие индентированные (два пробела) строки/пустые строки до следующей строки `- ` (следующая задача), заголовка `## ` (следующий квадрант) или строки с нулевой индентацией, не являющейся пустой. Для однострочной задачи блок = одна строка.
2. Удалить весь блок из исходного квадранта (с одной пустой строкой-разделителем рядом, чтобы не оставалось «дыры»).
3. Дописать обёрнутый в GFM-strikethrough блок в конец секции `Done`:
   - Однострочная задача: `- ~~[важность: X, срочность: Y] <текст> #tags~~ (YYYY-MM-DD)`.
   - Многострочная задача: `~~` ставится перед первой содержательной строкой блока (после `- `), и закрывающий `~~` — после тегов в последней строке. Индентация сохраняется. Дата `(YYYY-MM-DD)` следует за закрывающим `~~`.
4. Сохранить файл.

Это критично по архитектурному принципу скилла (см. раздел 1): задача «закрыта» с точки зрения трекера до того, как началась работа над реализацией.

### Шаг 5. Вопрос про брейншторм

Claude спрашивает через `AskUserQuestion`:

> «Задача перенесена в Done. Запускать `superpowers:brainstorming` для продумывания реализации?»

Варианты ответа:

- **`Да, запустить brainstorming`** → Claude немедленно вызывает `Skill` tool с аргументом `superpowers:brainstorming`. Дальнейшая работа идёт в рамках того скилла. Скилл `tasks` своё отработал.
- **`Нет, сам разберусь`** → Claude отвечает коротко `Ок, приступаю.` и либо сам начинает действовать в обычном режиме, либо ждёт, пока пользователь переключит Claude Code в plan mode. Никакого дополнительного workflow скилл `tasks` не навязывает — на усмотрение пользователя.

После ответа на этот вопрос скилл `tasks` завершается.

## 6. Размещение скилла

Файл скилла — `home/.claude/skills/tasks/SKILL.md`. Симлинкается в `~/.claude/skills/tasks/SKILL.md` через `sync.sh` (по паттерну существующих скиллов в репозитории, например `notebooklm-youtube-skill`, `youtube-search`).

Frontmatter скилла должен корректно описывать назначение и триггеры, чтобы команды `/add` и `/next` распознавались как точки входа.

## 7. Что осознанно НЕ входит в скоуп

Эти вещи намеренно исключены — если понадобятся, добавим позже.

- **Фильтрация `/next` по тегам** (`/next tech`, `/next product`). Теги остаются только для визуальной ориентации.
- **Учёт контекста репы** в `/next` (последние коммиты, изменённые файлы). Логика выбора — чисто по квадрантам.
- **Промежуточные статусы** (`In Progress`, `Blocked`). Только `Backlog` (4 квадранта) и `Done`.
- **Effort / оценка трудозатрат**. Только важность и срочность.
- **Чистка `Done`**. Растёт бесконечно, чистка — ручная, при необходимости.
- **Напоминания, дедлайны, даты добавления**. Дата ставится только при переносе в `Done`.
- **Редактирование существующих задач через скилл**. Правишь `TASKS.md` руками.
- **Удаление задач без выполнения**. Тоже руками — или через `/next` + немедленное «отменил», но это на усмотрение пользователя вне скоупа скилла.

## 8. Критерии успеха

- `/add <формулировка>` корректно создаёт `TASKS.md`, если его не было.
- `/add` на размытой формулировке задаёт уточняющие вопросы, а не ставит задачу «в лоб».
- `/add` на ясной формулировке доходит до подтверждения оценки за один шаг (без лишних вопросов).
- Оценка показывается через `AskUserQuestion`, пользователь может поправить в свободной форме.
- Задача попадает в правильный квадрант согласно таблице из раздела 3.
- `/next` на отсутствующем `TASKS.md` сообщает об этом и не создаёт файл.
- `/next` на пустых квадрантах корректно сообщает, что задач нет.
- `/next` набирает кандидатов строго по порядку квадрантов.
- Выбор задачи в `/next` происходит через `AskUserQuestion` с кнопками по каждой задаче.
- Выбранная задача немедленно переносится в `Done` со сегодняшней датой.
- Вопрос про brainstorming задаётся через `AskUserQuestion`; на «да» запускается `superpowers:brainstorming` через `Skill` tool.
