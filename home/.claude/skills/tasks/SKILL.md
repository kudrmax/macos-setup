# Tasks — менеджер задач в TASKS.md

## Назначение

Лёгкий трекер задач проекта. Файл `TASKS.md` лежит в корне репозитория. Скилл предоставляет две точки входа:

- **`/add <формулировка>`** — оценить задачу по матрице Эйзенхауэра (важность × срочность), подтвердить у пользователя, положить в нужный квадрант.
- **`/next`** — предложить топ-3 следующих задач, после выбора **немедленно** перенести в `Done` и спросить, запускать ли `superpowers:brainstorming`.

**Принцип:** скилл делает своё дело и забывается. Никаких промежуточных статусов. Как только пользователь выбрал задачу в `/next` — она сразу в `Done`, даже до начала реализации. Это сделано сознательно, чтобы Claude не держал в контексте «не забыть в конце закрыть задачу».

## Определение намерения

Скилл вызывается слэш-командами `/add` и `/next`. Команда передаёт управление в скилл с явным намерением:

- Если намерение `add` — выполняй раздел **«Поведение /add»** ниже. Аргументы пользователя — это исходная формулировка задачи.
- Если намерение `next` — выполняй раздел **«Поведение /next»** ниже. Аргументы не используются.

Если намерение не задано явно — попытайся определить его из сообщения пользователя.

## Формат TASKS.md

Файл лежит в корне проекта (`<repo-root>/TASKS.md`). Структура фиксирована:

```markdown
# Tasks

## Важно и срочно

- [важность: high, срочность: high] Починить падение pipeline при пустом .srt #tech #bug

## Важно, не срочно

- [важность: high, срочность: med] Вынести отбор target'ов в отдельный use case #tech #refactor
- [важность: med, срочность: low] Expose meaning errors via API #tech #product

## Не важно, срочно

- [важность: low, срочность: high] Обновить README — скриншоты устарели перед демо #docs

## Не важно, не срочно

- [важность: low, срочность: low] Поиграться со stanza вместо spaCy #tech #experiment

## Done

- ~~[важность: high, срочность: high] Настроить docker-compose для прод-сборки #tech~~ (2026-04-05)
```

**Правила формата:**

- Все 4 рабочих квадранта + секция `Done` присутствуют всегда, даже пустые. Порядок секций как выше — он значимый.
- Задача — это **блок** в markdown-списке. Минимум одна строка, может быть многострочной.
- **Однострочная задача:** `- [важность: <level>, срочность: <level>] <текст задачи> #tag1 #tag2`, где `<level>` ∈ {`high`, `med`, `low`}.
- **Многострочная задача** (когда пользователь прислал текст с контекстом/абзацами): первая строка `- [важность: <level>, срочность: <level>] <первая строка текста>`, последующие строки и абзацы индентируются **двумя пробелами** (markdown-list continuation). Между абзацами внутри задачи — пустая строка с индентацией или без. Теги ставятся в конце **последней** строки блока задачи.
- **Критично:** текст задачи сохраняется **дословно** как прислал пользователь. Никакой суммаризации или схлопывания в «заголовок». Задача с контекстом записывается с контекстом.
- Между блоками задач в одном квадранте — **одна пустая строка**.
- Префикс `[важность: X, срочность: Y]` обязателен. Квадрант — это **огрубление** оценки (см. §3), а префикс хранит полную детализацию для истории и возможной сортировки.
- Теги опциональны (0, 1 или несколько). Стандартные теги: `#tech`, `#product`, `#bug`, `#refactor`, `#docs`, `#experiment`. Свободные теги тоже допустимы.
- Новые задачи в `/add` дописываются **в конец** соответствующего квадранта (новые снизу).
- При переносе в `Done` **весь блок задачи** (включая многострочный контекст) оборачивается в GFM-strikethrough. Для многострочных задач: `~~` перед первым символом блока, `~~` после последнего символа тегов, дата `(YYYY-MM-DD)` после закрывающего `~~`. Префикс оценки сохраняется внутри strikethrough.
- Секция `Done` растёт вниз бесконечно. Скилл её НЕ чистит.

## Матрица приоритизации

| Важность | Срочность | Квадрант |
|---|---|---|
| `high` или `med` | `high` | Важно и срочно |
| `high` или `med` | `med` или `low` | Важно, не срочно |
| `low` | `high` | Не важно, срочно |
| `low` | `med` или `low` | Не важно, не срочно |

**Почему `med` идёт на «важную» сторону:** классический Эйзенхауэр бинарен («важно/не важно»), а мы держим три уровня для нюанса. Только явно низкая значимость попадает в «не важно». `med` — это «важно, но не критично», его место в Q1/Q2. Различие `high` vs `med` внутри одного квадранта сохраняется в префиксе `[важность: X, срочность: Y]` и может использоваться для сортировки или просто как справка.

**Как оценивать:**

- **Важность** — влияет ли задача на ключевые цели проекта. Если в корне есть `CLAUDE.md`, используй его как контекст целей. `high` = напрямую бьёт по core-value или core-workflow. `med` = заметно улучшает продукт/devex/архитектуру, но не блокирует. `low` = эксперимент, полировка, «когда-нибудь».
- **Срочность** — есть ли временное давление. Признаки: маркеры «горит», «блокирует», «прод-баг», упоминания дедлайнов, внешних демо/встреч. `high` = блокер или явный внешний дедлайн. `med` = хочется скоро, но без жёсткого срока. `low` = нет временного давления.

## Поведение /add

Аргументы: исходная формулировка задачи от пользователя (`$ARGUMENTS` из слэш-команды).

### Шаг 0. Распознавание нескольких задач

`/add` может принять несколько задач за один вызов. Определяй это **семантически**, а не только по формальным разделителям — пользователь может закинуть несколько задач подряд в одном абзаце без явных маркеров.

**Явные признаки** (если есть — точно несколько задач):
- Разделение по новой строке (каждая задача с новой строки).
- Явные разделители: `;`, `|`, нумерованный список (`1.`, `2.`), буллеты (`-`, `*`).
- Явная формулировка списка в прозе: «добавь задачу X и задачу Y», «нужно сделать: A, B, C».

**Семантические признаки** (нет формальных маркеров, но смысл указывает на несколько задач):
- Текст описывает несколько независимых проблем/работ, не связанных между собой причинно-следственно.
- Между предложениями резкая смена темы/объекта/компонента (одна про БД, следующая про линтер — это две разные задачи, даже если они в одном абзаце).
- Каждая часть может быть самостоятельно завершена без другой.

Если сомневаешься — один это запрос или несколько — спроси пользователя через `AskUserQuestion`:

> «Вижу потенциально <N> задач: <кратко перечисли>. Разбить на отдельные или записать как одну?»
> Варианты: `Разбить на N` / `Одна задача`.

Если задача одна — выполняй шаги 1–5 как обычно.

Если задач несколько — обработай каждую **последовательно** через полный цикл: Шаг 1 (проверка файла) выполняется один раз перед первой задачей, затем для каждой задачи по очереди Шаги 2–5. После записи одной задачи сразу переходи к следующей.

Уточняющие вопросы (Шаг 2) и подтверждение оценки (Шаг 4) задавай по каждой задаче отдельно — не объединяй в один `AskUserQuestion` для нескольких задач сразу.

После обработки **всех** задач из списка выведи итоговую сводку одним сообщением вместо индивидуального `Добавлено в «<квадрант>».` после каждой:

```
Добавлено:
1. «<задача 1>» → Важно и срочно
2. «<задача 2>» → Не важно, не срочно
3. «<задача 3>» → Важно, не срочно
```

### Шаг 1. Проверка TASKS.md

Проверь наличие `TASKS.md` в корне текущего проекта (рабочая директория Claude Code).

- Если файла нет — создай его со следующим пустым каркасом:

```markdown
# Tasks

## Важно и срочно

## Важно, не срочно

## Не важно, срочно

## Не важно, не срочно

## Done
```

- Если файл есть — продолжай.

### Шаг 2. Проверка ясности формулировки

Оцени, достаточно ли информации в аргументах для осмысленной оценки важности и срочности.

**Признаки недостаточной ясности:**
- Слишком коротко/общо («починить баг», «рефакторинг», «добавить фичу»).
- Непонятно о каком компоненте/файле/сценарии речь.
- Неясно, блокер это или «когда-нибудь».
- Непонятно, баг в проде или гипотетический edge case.

Если формулировка размытая — задавай уточняющие вопросы **по одному**:

- Close-ended вопросы (например, «это блокирует работу сейчас или можно отложить?») — используй tool `AskUserQuestion` с готовыми вариантами.
- Open-ended вопросы, требующие развёрнутого ответа («что именно ломается?») — задавай обычным текстом.

После сбора контекста переформулируй задачу в более конкретную и спроси через `AskUserQuestion`:

> «Переформулирую: *<новая формулировка>*. Ок?»
> Варианты: `Ок` / `Ещё поправить`

Цикл до подтверждения.

### Шаг 3. Оценка

**Главное правило обработки текста задачи:**

- **Переформулировать можно и часто полезно** — привести к связному виду, поправить грамматику, сделать фразы более чёткими. Пользователь может написать коряво или в потоке мысли — причесать это хорошо.
- **Сокращать и терять контекст — категорически нельзя.** Если пользователь прислал задачу с описанием «что ломается, где именно (какие файлы/компоненты/endpoints), почему, какой предлагаемый фикс, какие есть нюансы» — **весь этот контекст** должен оказаться в итоговой записи. НЕ схлопывай задачу в одно предложение-заголовок. НЕ выбрасывай упоминания конкретных файлов, таблиц БД, endpoint'ов, имён функций, причин инцидентов, альтернатив и т.д.
- **Проверка на себе:** прочитай свою переформулированную версию. Если из неё нельзя понять «что конкретно делать и почему», а из исходника — можно, значит ты потерял важное. Верни контекст.

**Что можно отбросить** (это шум, а не контекст):
- «Шумовые» нумерационные префиксы в начале: `#11 — `, `[1]`, `Задача 1:`, и т.п. — это внешняя нумерация пользователя, не часть задачи.
- Явные обращения «добавь в tasks», «запиши задачу», «нужно добавить» — это инструкция скилла, не содержание.

Если **оригинал уже хорошо сформулирован** — сохраняй дословно, не пытайся улучшать ради улучшения.

На основе финального текста проставь метаданные:
- **Важность:** `high` / `med` / `low`
- **Срочность:** `high` / `med` / `low`
- **Теги:** из стандартного набора (`#tech`, `#product`, `#bug`, `#refactor`, `#docs`, `#experiment`) и свободные, если явно напрашиваются.

Вычисли квадрант по таблице из раздела «Матрица приоритизации».

- **Важность:** `high` / `med` / `low`

- **Теги:** из стандартного набора (`#tech`, `#product`, `#bug`, `#refactor`, `#docs`, `#experiment`) и свободные, если явно напрашиваются.

### Шаг 4. Подтверждение оценки

Покажи пользователю результат одним сообщением и спроси через tool `AskUserQuestion`.

**Формат вывода (обязательный — не упрощать):**

```markdown
**Задача <N>/<total>** · _<квадрант>_

> <полный текст задачи, как будет записан в файл — со всеми абзацами, без сокращений>

- **Важность:** `<level>` — <краткое обоснование>
- **Срочность:** `<level>` — <краткое обоснование>
- **Теги:** `#tag1` `#tag2`
```

Заголовок `**Задача <N>/<total>**` показывай всегда: для одиночной задачи это `Задача 1/1`, для множественного `/add` — `Задача 1/4`, `Задача 2/4` и т.д.

Текст задачи показывай **полностью** в blockquote — даже если он на несколько абзацев. Blockquote корректно рендерит многострочный текст. Это гарантия, что то, что видит пользователь при подтверждении = то, что будет записано в файл.

После вывода — спроси через `AskUserQuestion`. Варианты: `Ок, записать` / `Поправить`.

Если `Поправить` — перейди в свободный диалог. Принимай правки в произвольной форме («срочность low», «убери #bug, добавь #refactor», «это в product, не tech», «сократи формулировку»), применяй, показывай обновлённую карточку в том же формате и снова спрашивай через `AskUserQuestion`. Цикл до подтверждения.

### Шаг 5. Запись в TASKS.md

Добавь задачу **в конец** нужного квадранта. Формат записи:

**Если задача однострочная** — одна строка:
```
- [важность: X, срочность: Y] <полный текст задачи> #tag1 #tag2
```

**Если задача многострочная** (несколько абзацев, содержит переводы строки) — используй markdown-list continuation: первая строка начинается с `- `, последующие строки индентируются двумя пробелами. Теги ставь в конце **последней** строки блока. Пример:

```
- [важность: med, срочность: low] Expose meaning errors via API

  Когда AI-генерация кандидата падает, текст ошибки сохраняется в candidate_meanings.error и пишется в логи воркера, но ни один API-endpoint его наружу не отдаёт. В UI видно только счётчики «failed», без причины.

  Фикс: добавить endpoint (GET /sources/{id}/meanings/failed, возвращающий [{candidate_id, error}]) и показать в UI. #tech #product
```

Первая строка после префикса `[важность: X, срочность: Y]` — это короткий «заголовок» задачи (как правило, первая смысловая фраза). Дальше, через пустую строку и двухпробельную индентацию — остальной контекст. Между блоками задач в одном квадранте оставляй **одну пустую строку**.

**Использование `Edit` tool:**

**Случай A — квадрант уже содержит задачи.** `old_string` — последняя строка последнего блока задач в этом квадранте (например, последняя строка текста с тегами). `new_string` — та же строка плюс пустая строка и новый блок задачи.

**Случай B — квадрант пуст.** `old_string` — заголовок квадранта плюс пустая строка плюс заголовок следующего квадранта (для уникальности). `new_string` — тот же заголовок, пустая строка, новый блок задачи, пустая строка, заголовок следующего квадранта.

Пример (квадрант «Важно, не срочно» пуст, вставляем многострочную задачу):
```
old_string: "## Важно, не срочно\n\n## Не важно, срочно"
new_string: "## Важно, не срочно\n\n- [важность: med, срочность: low] Expose meaning errors via API\n\n  Когда AI-генерация...\n\n  Фикс: добавить endpoint... #tech #product\n\n## Не важно, срочно"
```

Если пустой квадрант — **последний** рабочий квадрант перед `## Done` — используй `## Done` как нижний маркер в `old_string`.

**Подтверждение пользователю:**

- Если обрабатывалась **одна задача** — одна строка: `Добавлено в «<название квадранта>».`
- Если обрабатывалось **несколько задач** — **не** отвечай после каждой. Только после обработки всех покажи итоговую сводку в формате:

```markdown
**Добавлено:**

1. «<первая строка задачи 1>» → _<квадрант>_
2. «<первая строка задачи 2>» → _<квадрант>_
3. «<первая строка задачи 3>» → _<квадрант>_
```

Скилл завершён.

## Поведение /next

Аргументы не используются.

### Шаг 1. Проверка TASKS.md

- Если `TASKS.md` в корне проекта **отсутствует** — ответь коротко:
  > В проекте нет задач. Добавь первую через `/add <формулировка>`.

  **Файл НЕ создавай.** Завершись.

- Если `TASKS.md` есть, но все 4 рабочих квадранта пусты (независимо от того, пуста ли секция `Done`) — ответь:
  > Все задачи завершены. Добавь новые через `/add`.

  Завершись.

- Иначе — продолжай.

### Шаг 2. Сбор кандидатов

Прочитай `TASKS.md` и собери до **3 задач** строго по порядку квадрантов:

1. `Важно и срочно`
2. `Важно, не срочно`
3. `Не важно, срочно`
4. `Не важно, не срочно`

Внутри квадранта — в том порядке, в котором задачи записаны в файле (сверху вниз). Набирай первые 3 какие встретишь. Не смешивай и не переставляй. Логика строго детерминированная — никаких учётов тегов, последних коммитов или контекста репы.

### Шаг 3. Предложение пользователю

Покажи кандидатов пользователю одним сообщением и спроси через tool `AskUserQuestion`. Каждый кандидат — отдельный вариант ответа в `AskUserQuestion`. Дополнительный вариант — `Другое (вручную)`.

Для каждой задачи в preview показывай **первую строку** блока (после префикса `[важность: ..., срочность: ...]`) как заголовок. Если задача многострочная — полный контекст в preview НЕ вставляй, чтобы сообщение оставалось читаемым. Пользователь уже знает свои задачи, ему достаточно узнать их по первой фразе.

Формат сообщения-превью (перед вызовом `AskUserQuestion`):

```markdown
**Следующие задачи:**

1. <первая строка задачи 1> · _<квадрант 1>_ · `<теги>`
2. <первая строка задачи 2> · _<квадрант 2>_ · `<теги>`
3. <первая строка задачи 3> · _<квадрант 3>_ · `<теги>`
```

Если кандидатов меньше 3 — показывай сколько есть.

**Обработка «Другое (вручную)»:** попроси пользователя назвать задачу свободной формулировкой. Ищи её в `TASKS.md` по нечёткому совпадению (подстрока, игнорируя регистр и знаки препинания) — ищи и в первой строке блока, и в многострочном контексте. Если не находишь — переспроси один раз. Если снова нет — ответь «Не смог найти задачу в TASKS.md» и предложи выбрать из изначальных трёх кандидатов.

> **⛔ СТОП-ЯКОРЬ — прочитай перед тем, как обработать ответ пользователя.**
>
> Когда пользователь выберет задачу, твоё **первое и единственное** действие — Шаг 4 (перенос в Done). НЕ исследуй код, НЕ запускай Agent, НЕ обсуждай реализацию, НЕ делай НИЧЕГО кроме редактирования TASKS.md. Содержание задачи (баг, фича, рефакторинг) — это приманка для «режима реализации». Игнорируй её. Сначала — Done, потом — Шаг 5.

### Шаг 4. Перенос в Done

**Немедленно** после выбора задачи, до любого обсуждения реализации:

1. Прочитай `TASKS.md`.
2. **Определи границы блока задачи.** Блок задачи начинается со строки `- [важность: X, срочность: Y] ...` и включает:
   - эту первую строку,
   - все последующие **индентированные** (два пробела) строки и пустые строки с такой же индентацией, пока не встретится следующая строка, начинающаяся с `- ` (следующая задача), или заголовок `## ` (следующий квадрант), или строка с нулевой индентацией не являющаяся пустой.
   - Если задача однострочная — блок = одна строка.
3. **Удали блок** через `Edit` tool: `old_string` — весь блок от первой строки `- [...]` до последней строки с тегами включительно, плюс одна пустая строка-разделитель после него (если задача не последняя в квадранте) или перед ним (если последняя). `new_string` — пустая строка. Если блок совпадает по тексту с другим — расширь `old_string` соседними строками для уникальности.
4. **Оборачивай весь блок в strikethrough и добавь в Done.** Формат:
   - Однострочная задача: `- ~~[важность: X, срочность: Y] <текст> #tags~~ (YYYY-MM-DD)`
   - Многострочная задача: оборачиваем `~~` вокруг всего блока, включая индентированные строки. Пример:
     ```
     - ~~[важность: med, срочность: low] Expose meaning errors via API

       Когда AI-генерация кандидата падает...

       Фикс: добавить endpoint... #tech #product~~ (2026-04-09)
     ```
   - Дата — сегодняшняя.
5. Допиши обёрнутый блок в конец секции `Done` (по аналогии с Шагом 5 команды `/add` — случай A или B в зависимости от того, пуста ли секция Done).
6. Сохрани файл (`Edit` tool сам сохраняет при успешной записи).

Это критично: задача должна оказаться в `Done` **до** того, как начнётся работа над реализацией. Принцип скилла — «отработал и забыл».

### Шаг 5. Вопрос про брейншторм

Спроси пользователя через tool `AskUserQuestion`:

> «Задача перенесена в Done. Запускать `superpowers:brainstorming` для продумывания реализации?»

Варианты:
- **`Да, запустить brainstorming`** → немедленно вызови tool `Skill` с аргументом `superpowers:brainstorming`. Передай исходную формулировку выбранной задачи как контекст для брейншторма. Далее работа идёт в рамках того скилла — скилл `tasks` своё отработал.
- **`Нет, сам разберусь`** → ответь коротко: `Ок, приступаю.` На этом скилл `tasks` завершается. Что делать дальше — зависит от пользователя: он либо сам переключит Claude Code в plan mode, либо даст прямое указание действовать. Никакой дополнительный workflow скилл `tasks` не навязывает.

Скилл завершён.
